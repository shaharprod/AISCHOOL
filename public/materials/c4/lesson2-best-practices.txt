Best Practices ל-System Instructions
=====================================

שיטות עבודה מומלצות - המדריך המלא
===================================

1. מבנה וסדר
=============

סדר מומלץ:
-----------
1. הגדרת תפקיד (Role)
2. הגדרת מטרה (Goal)
3. כללי התנהגות (Rules)
4. פורמט תשובה (Format)
5. דוגמאות (Examples)

דוגמה טובה:
```
אתה מפתח Python מנוסה. (תפקיד)

המטרה שלך היא לעזור למשתמשים לכתוב קוד Python טוב. (מטרה)

כללי קוד:
- עקוב אחר PEP 8
- השתמש ב-type hints
- כתוב docstrings (כללים)

פורמט תשובה:
1. קוד מלא
2. הסבר קצר
3. נקודות חשובות (פורמט)

דוגמה:
שאלה: "כתוב פונקציה לחישוב ממוצע"
תשובה: [תשובה מושלמת] (דוגמה)
```

2. בהירות ופשטות
==================

✅ עשה:
-------
• השתמש בשפה פשוטה וברורה
• כתוב משפטים קצרים
• השתמש ברשימות נקודות
• הימנע ממילים מורכבות

❌ אל תעשה:
------------
• אל תשתמש בז'רגון מיותר
• אל תכתוב משפטים ארוכים מדי
• אל תערבב נושאים
• אל תהיה מעורפל

דוגמה טובה:
```
אתה מפתח Python.
כתוב קוד נקי.
עקוב אחר PEP 8.
```

דוגמה רעה:
```
אתה צריך להיות מפתח Python מקצועי שיודע לכתוב קוד
שנראה טוב ופועל כמו שצריך וניתן לקריאה וניתן לתחזוקה
ועקוב אחר כל הכללים והתקנים...
```

3. ספציפיות
=============

✅ עשה:
-------
• ציין בדיוק מה אתה רוצה
• תן דוגמאות קונקרטיות
• הגדר גבולות ברורים
• ציין מה אסור

❌ אל תעשה:
------------
• אל תהיה מעורפל
• אל תשאיר מקום לפרשנות
• אל תכתוב "תמיד תן תשובות טובות"

דוגמה טובה:
```
תמיד הוסף type hints לפונקציות.
עקוב אחר PEP 8.
כתוב docstrings עם Google style.
```

דוגמה רעה:
```
כתוב קוד טוב.
```

4. אורך מתאים
==============

אורך מומלץ:
------------
• מינימום: 20-30 מילים
• אידיאלי: 50-200 מילים
• מקסימום: 300-400 מילים

למה?
-----
• קצר מדי - לא מספיק מידע
• ארוך מדי - המודל מתעלם או מתקשה לעקוב
• אידיאלי - מספיק מידע בלי עומס

דוגמה קצרה מדי:
```
אתה מפתח Python.
```

דוגמה ארוכה מדי:
```
אתה מפתח Python מנוסה עם 10+ שנות ניסיון שעבד על
פרויקטים גדולים ומורכבים וצבר ניסיון רב בכתיבת קוד
נקי וקריא וניתן לתחזוקה ועקב אחר כל התקנים והכללים
והשתמש בכלים מתקדמים וטכניקות מודרניות...
[עוד 500 מילים...]
```

דוגמה אידיאלית:
```
אתה מפתח Python מנוסה עם 10+ שנות ניסיון.

כללי קוד:
- עקוב אחר PEP 8
- השתמש ב-type hints
- כתוב docstrings מפורטים
- הוסף הערות במקומות מורכבים

פורמט תשובה:
1. קוד מלא
2. הסבר קצר
3. נקודות חשובות
```

5. עקביות
==========

✅ עשה:
-------
• השתמש באותה שפה וסגנון
• שמור על מבנה עקבי
• השתמש באותם מונחים
• עקוב אחר אותם כללים

❌ אל תעשה:
------------
• אל תשנה סגנון באמצע
• אל תשתמש במונחים סותרים
• אל תכתוב כללים סותרים

דוגמה טובה:
```
תמיד תן תשובות קצרות (2-3 משפטים).
אם המשתמש מבקש, הרחב עם פרטים.
```

דוגמה רעה:
```
תמיד תן תשובות קצרות.
תמיד תן הסברים מפורטים.
```

6. דוגמאות
===========

למה דוגמאות חשובות?
---------------------
• מראות למה אתה מתכוון
• מפשטות הבנה
• משפרות consistency
• עוזרות למודל ללמוד

איך לכתוב דוגמאות טובות:
--------------------------
• השתמש בדוגמאות ריאליות
• הצג input ו-output
• הסבר למה הדוגמה טובה
• הוסף כמה דוגמאות

דוגמה טובה:
```
דוגמה לשאלה: "כתוב פונקציה לחישוב ממוצע"

תשובה טובה:
```python
def calculate_average(numbers: list[float]) -> float:
    """
    Calculate the average of a list of numbers.
    
    Args:
        numbers: List of numbers to calculate average for
        
    Returns:
        Average of the numbers
    """
    if not numbers:
        raise ValueError("List cannot be empty")
    return sum(numbers) / len(numbers)
```

למה זה טוב:
- יש type hints
- יש docstring מפורט
- יש error handling
- הקוד נקי וקריא
```

7. הימנעות מהוראות סותרות
============================

טעויות נפוצות:
---------------
❌ "תמיד תן תשובות קצרות" + "תמיד תן הסברים מפורטים"
❌ "תמיד השתמש במונחים טכניים" + "תמיד הסבר במילים פשוטות"
❌ "תמיד תן דוגמאות" + "אל תכלול דוגמאות"

פתרון:
-------
✅ "תמיד תן תשובות קצרות (2-3 משפטים). אם המשתמש מבקש, הרחב."
✅ "השתמש במונחים טכניים, אבל הסבר אותם במילים פשוטות."
✅ "תמיד תן דוגמה אחת לפחות, אלא אם המשתמש מבקש אחרת."

8. שימוש בתנאים
================

מתי להשתמש בתנאים:
-------------------
• כשצריך התנהגות שונה לפי הקשר
• כשצריך גמישות
• כשצריך לטפל ב-edge cases

דוגמה טובה:
```
אם המשתמש שואל שאלה טכנית:
  - תן תשובה מפורטת עם דוגמאות קוד
  - השתמש במונחים טכניים

אם המשתמש שואל שאלה כללית:
  - תן תשובה קצרה וברורה
  - השתמש במילים פשוטות
```

9. עדכון ושיפור
================

מתי לעדכן:
----------
• כשהמודל לא עוקב אחר ההוראות
• כשצריך התנהגות שונה
• כשמקבלים feedback
• כשמשנים use case

איך לעדכן:
-----------
• בדוק מה לא עובד
• שפר בהדרגה
• בדוק שוב
• תיעד שינויים

10. בדיקה ואופטימיזציה
========================

שיטת בדיקה:
------------
1. כתוב System Instructions
2. בדוק עם שאלות שונות
3. מדוד ביצועים
4. שפר בהדרגה
5. בדוק שוב

שאלות לבדיקה:
--------------
• שאלות פשוטות
• שאלות מורכבות
• שאלות קצה (edge cases)
• שאלות לא צפויות

מדדי ביצועים:
-------------
• האם המודל עוקב אחר ההוראות?
• האם התשובות עקביות?
• האם הפורמט נכון?
• האם הטון מתאים?

11. A/B Testing
================

למה לעשות A/B Testing:
-----------------------
• למצוא את הגרסה הטובה ביותר
• להבין מה עובד
• לשפר בהדרגה

איך לעשות:
-----------
1. כתוב שתי גרסאות
2. בדוק עם אותן שאלות
3. השווה תוצאות
4. בחר את הטובה ביותר

דוגמה:
```
גרסה A (קצרה):
אתה מפתח Python. כתוב קוד נקי.

גרסה B (מפורטת):
אתה מפתח Python מנוסה.
כתוב קוד נקי וקריא.
עקוב אחר PEP 8.
השתמש ב-type hints.
```

12. תיעוד
==========

למה לתעד:
---------
• לזכור מה עבד
• לשתף עם אחרים
• לשפר בעתיד

מה לתעד:
---------
• גרסה של System Instructions
• תאריך עדכון
• מה שונה ולמה
• תוצאות בדיקות

דוגמה:
```
גרסה 1.0 - 2026-01-28
- גרסה ראשונית
- בדיקה בסיסית עברה

גרסה 1.1 - 2026-01-29
- הוספתי דוגמאות
- שיפרתי בהירות
- בדיקה מפורטת עברה
```

13. שימוש חוזר
===============

למה ליצור templates:
---------------------
• חיסכון בזמן
• עקביות
• איכות

איך ליצור:
-----------
• שמור System Instructions מוצלחות
• צור templates לפי קטגוריות
• התאם לפי צורך

דוגמה:
```
Template: מפתח תוכנה
[תפקיד] מנוסה ב-[שפה/טכנולוגיה].

כללי קוד:
- [כלל 1]
- [כלל 2]
- [כלל 3]

פורמט:
[פורמט]
```

14. שיתוף ולמידה
=================

למה לשתף:
----------
• ללמוד מאחרים
• לשפר יחד
• לבנות קהילה

איך לשתף:
----------
• שתף System Instructions מוצלחות
• בקש feedback
• למד מטעויות
• שפר יחד

טיפים נוספים
==============

1. התחל פשוט
-------------
• התחל עם גרסה בסיסית
• שפר בהדרגה
• אל תתחיל מורכב מדי

2. בדוק לעיתים קרובות
-----------------------
• בדוק אחרי כל שינוי
• מדוד ביצועים
• שפר בהתבסס על תוצאות

3. היה גמיש
-------------
• התאם לפי צורך
• אל תהיה נוקשה מדי
• אפשר גמישות כשצריך

4. למד מטעויות
---------------
• זהה מה לא עובד
• למד למה
• שפר בהתאם

5. שמור על פשטות
------------------
• אל תסבך מיותר
• פשטות היא כוח
• פחות יכול להיות יותר
