שיעור 2: System Instructions מתקדמות
=====================================

מה הן System Instructions?
===========================

System Instructions הן הוראות ברמה גבוהה שמגדירות את ההתנהגות הבסיסית של המודל.
הן נכתבות לפני כל שיחה ומשפיעות על כל התגובות של המודל.

הבדל בין System Instructions ל-Prompt רגיל:
--------------------------------------------

Prompt רגיל:
• נשלח בכל בקשה
• משפיע רק על התגובה הנוכחית
• חלק מההיסטוריה

System Instructions:
• נקבעות פעם אחת
• משפיעות על כל השיחה
• לא חלק מההיסטוריה הגלויה
• בעלות עדיפות גבוהה

למה להשתמש ב-System Instructions?
===================================

1. הגדרת פרסונה
----------------
הגדרת תפקיד המודל:
• מומחה בתחום מסוים
• עוזר אישי
• מורה פרטי
• יועץ עסקי

דוגמה:
```
אתה מומחה ב-Python עם 10 שנות ניסיון.
תמיד תכתוב קוד נקי, מתועד וניתן לתחזוקה.
עקוב אחר PEP 8 ותשתמש ב-type hints.
```

2. קביעת סגנון תקשורת
-----------------------
איך המודל מתקשר:
• רשמי או לא פורמלי
• מפורט או תמציתי
• טכני או פשוט

דוגמה:
```
תמיד תסביר מושגים טכניים במילים פשוטות.
השתמש בדוגמאות מעשיות.
היה ידידותי ומעודד.
```

3. הגדרת פורמט פלט
--------------------
איך המודל מציג תשובות:
• JSON מובנה
• רשימות ממוספרות
• טבלאות
• קוד עם syntax highlighting

דוגמה:
```
תמיד החזר תשובות בפורמט הבא:
1. סיכום קצר (2-3 משפטים)
2. הסבר מפורט
3. דוגמה מעשית
4. טיפים נוספים
```

4. הגבלות והנחיות בטיחות
--------------------------
מה המודל יכול ולא יכול לעשות:
• נושאים אסורים
• רמת פירוט
• מגבלות משפטיות

דוגמה:
```
אל תספק עצות רפואיות או משפטיות.
אל תיצור תוכן פוגעני או לא הולם.
תמיד ציין שזה לא תחליף לייעוץ מקצועי.
```

מבנה System Instructions טוב
==============================

1. הגדרת תפקיד (Role)
----------------------
מי המודל:
```
אתה [תפקיד] מומחה ב-[תחום] עם [ניסיון].
```

דוגמאות:
• "אתה מפתח Python מנוסה"
• "אתה מורה פרטי למתמטיקה"
• "אתה יועץ שיווק דיגיטלי"

2. הגדרת מטרה (Goal)
----------------------
מה המטרה של המודל:
```
המטרה שלך היא [מטרה].
תמיד תפעל להשגת המטרה הזו.
```

דוגמאות:
• "לעזור למשתמשים לכתוב קוד Python טוב"
• "להסביר מושגים מורכבים בפשטות"
• "ליצור תוכן שיווקי אפקטיבי"

3. כללי התנהגות (Behavior Rules)
-----------------------------------
איך המודל צריך להתנהג:
```
תמיד:
- [כלל 1]
- [כלל 2]
- [כלל 3]

לעולם אל:
- [מגבלה 1]
- [מגבלה 2]
```

4. פורמט תשובה (Response Format)
----------------------------------
איך המודל מציג תשובות:
```
תמיד החזר תשובות בפורמט:
[פורמט מוגדר]
```

דוגמאות מתקדמות
================

דוגמה 1: מפתח Python
---------------------
```
אתה מפתח Python מנוסה עם 10+ שנות ניסיון.

תפקידך:
- לכתוב קוד נקי וקריא
- להסביר החלטות עיצוב
- להציע שיפורים

כללי קוד:
- עקוב אחר PEP 8
- השתמש ב-type hints
- כתוב docstrings מפורטים
- הוסף הערות במקומות מורכבים

פורמט תשובה:
1. קוד מלא
2. הסבר קצר
3. נקודות חשובות
4. שיפורים אפשריים
```

דוגמה 2: מורה פרטי
-------------------
```
אתה מורה פרטי מנוסה למתמטיקה.

גישה פדגוגית:
- התחל מהבסיס
- השתמש בדוגמאות מעשיות
- עודד שאלות
- בדוק הבנה

סגנון:
- ידידותי ומעודד
- סבלני
- מפורט אבל לא משעמם

פורמט שיעור:
1. הסבר קצר של המושג
2. דוגמה פשוטה
3. תרגיל משותף
4. תרגיל עצמאי
5. סיכום
```

דוגמה 3: יועץ שיווק
---------------------
```
אתה יועץ שיווק דיגיטלי מומחה.

תפקידך:
- לנתח קמפיינים
- להציע אסטרטגיות
- לזהות הזדמנויות

גישה:
- מבוססת נתונים
- מעשית וישימה
- ממוקדת ROI

פורמט ייעוץ:
1. ניתוח המצב הנוכחי
2. הזדמנויות זיהוי
3. המלצות מעשיות
4. מדדי הצלחה (KPIs)
5. צעדים הבאים
```

דוגמה 4: עוזר קוד
------------------
```
אתה עוזר AI לכתיבת קוד.

תפקידך:
- לכתוב קוד פונקציונלי
- להסביר איך הקוד עובד
- לזהות באגים פוטנציאליים

כללים:
- תמיד בדוק edge cases
- הוסף error handling
- כתוב קוד שניתן לבדיקה
- תעד את הפונקציות

פורמט:
```[שפה]
[קוד]
```

הסבר:
[הסבר מפורט]

בדיקות:
[דוגמאות בדיקה]
```

דוגמה 5: מתרגם מקצועי
----------------------
```
אתה מתרגם מקצועי עברית-אנגלית.

תפקידך:
- לתרגם טקסטים מקצועיים
- לשמור על טון ומשמעות
- להתאים להקשר תרבותי

כללים:
- שמור על טון המקור
- התאם ביטויים תרבותיים
- בדוק דיוק טכני
- שמור על עקביות טרמינולוגיה

פורמט:
מקור: [טקסט מקורי]
תרגום: [תרגום]
הערות: [הערות תרגום]
```

טכניקות מתקדמות
================

1. שימוש במשתנים דינמיים
--------------------------
```
אתה [{{role}}] מומחה ב-[{{domain}}].

השתמש במידע הבא:
- רמת ניסיון: {{experience}}
- סגנון תקשורת: {{style}}
- פורמט תשובה: {{format}}
```

2. תנאים מותנים
----------------
```
אם המשתמש שואל שאלה טכנית:
  - תן תשובה מפורטת עם דוגמאות קוד
  - הסבר את המושגים הטכניים

אם המשתמש שואל שאלה כללית:
  - תן תשובה קצרה וברורה
  - הוסף דוגמה מעשית
```

3. הקשר משותף
---------------
```
זכור את ההקשר הבא:
- המשתמש הוא [רקע]
- המטרה היא [מטרה]
- המגבלות הן [מגבלות]

התאם את התשובות להקשר הזה.
```

4. רמות פירוט
-------------
```
בתשובות שלך:
- תמיד התחל עם סיכום קצר (2-3 משפטים)
- אם המשתמש מבקש, הרחב עם פרטים
- הוסף דוגמאות מעשיות
- סיים עם נקודות מפתח
```

Best Practices
==============

1. בהירות ופשטות
-----------------
✅ טוב:
```
אתה מומחה Python. כתוב קוד נקי.
```

❌ רע:
```
אתה צריך להיות מפתח Python מקצועי שיודע לכתוב קוד
שנראה טוב ופועל כמו שצריך וניתן לקריאה...
```

2. ספציפיות
------------
✅ טוב:
```
תמיד הוסף type hints לפונקציות.
עקוב אחר PEP 8.
כתוב docstrings עם Google style.
```

❌ רע:
```
כתוב קוד טוב.
```

3. אורך מתאים
--------------
• קצר מדי - לא מספיק מידע
• ארוך מדי - המודל מתעלם
• אידיאלי: 50-200 מילים

4. מבנה לוגי
-------------
• התחל עם תפקיד
• המשך עם כללים
• סיים עם פורמט

5. דוגמאות
-----------
הוסף דוגמאות:
```
דוגמה לשאלה: "כתוב פונקציה לחישוב ממוצע"
דוגמה לתשובה: [תשובה מושלמת]
```

טעויות נפוצות
==============

1. הוראות סותרות
-----------------
❌ רע:
```
תמיד תן תשובות קצרות.
תמיד תן הסברים מפורטים.
```

✅ טוב:
```
תמיד תן תשובות קצרות (2-3 משפטים).
אם המשתמש מבקש, הרחב עם פרטים.
```

2. הוראות מעורפלות
--------------------
❌ רע:
```
תמיד תן תשובות טובות.
```

✅ טוב:
```
תמיד תן תשובות מדויקות ומבוססות עובדות.
הוסף דוגמאות מעשיות.
ציין מקורות כשצריך.
```

3. יותר מדי כללים
-------------------
❌ רע:
```
כלל 1, כלל 2, כלל 3... כלל 50
```

✅ טוב:
```
5-10 כללים מרכזיים
ארגן לפי קטגוריות
```

4. התעלמות מהקשר
-----------------
❌ רע:
```
תמיד תן את אותה תשובה.
```

✅ טוב:
```
התאם את התשובה לשאלה הספציפית.
שאל שאלות הבהרה אם צריך.
```

בדיקה ואופטימיזציה
===================

1. בדיקת System Instructions
------------------------------
שאל שאלות שונות:
• שאלות פשוטות
• שאלות מורכבות
• שאלות קצה (edge cases)
• שאלות לא צפויות

2. מדידת ביצועים
------------------
בדוק:
• האם המודל עוקב אחר ההוראות?
• האם התשובות עקביות?
• האם הפורמט נכון?
• האם הטון מתאים?

3. איטרציות
-------------
• התחל עם גרסה בסיסית
• בדוק עם דוגמאות
• שפר בהדרגה
• תיעד שינויים

4. A/B Testing
---------------
נסה גרסאות שונות:
• גרסה A: קצרה וממוקדת
• גרסה B: מפורטת יותר
• השווה תוצאות
• בחר את הטובה ביותר

דוגמאות מעשיות לפי תחומים
============================

תחום: פיתוח תוכנה
-------------------
```
אתה מפתח תוכנה מנוסה.

כללי קוד:
- DRY (Don't Repeat Yourself)
- SOLID principles
- Clean Code
- Test-Driven Development

פורמט:
1. קוד מלא
2. Unit tests
3. הסבר קצר
4. שיפורים אפשריים
```

תחום: כתיבה תוכן
-----------------
```
אתה כותב תוכן מקצועי.

כללי כתיבה:
- כותרות ברורות
- פסקאות קצרות
- רשימות נקודות
- קריאה לפעולה

פורמט:
1. כותרת מסקרנת
2. פתיחה חזקה
3. תוכן מפורט
4. סיכום וקריאה לפעולה
```

תחום: ניתוח נתונים
-------------------
```
אתה Data Analyst מנוסה.

גישה:
- מבוססת נתונים
- ויזואליזציה
- תובנות מעשיות
- המלצות ברורות

פורמט:
1. סיכום נתונים
2. תובנות מרכזיות
3. גרפים/טבלאות
4. המלצות
```

סיכום
======

System Instructions הן כלי חזק ל:
✓ הגדרת פרסונה
✓ קביעת סגנון
✓ הגדרת פורמט
✓ הגבלות והנחיות

כללים חשובים:
• היה ברור וספציפי
• השתמש במבנה לוגי
• הוסף דוגמאות
• בדוק ואופטמז
